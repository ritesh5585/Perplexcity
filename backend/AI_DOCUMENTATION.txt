1. PROJECT OVERVIEW
================================================================================

This project is a Perplexity-style AI search engine. Users send a question via 
API, and the AI agent:
  1. Reads the question
  2. Decides if it needs to search the web (using Tavily search)
  3. Searches the web if needed
  4. Generates a comprehensive answer using Groq's Mixtral LLM
  5. Returns the answer to the user

TECH STACK:
  • Backend Framework: Express.js v5
  • Database: MongoDB (via Mongoose)
  • AI/LLM: Groq (Mixtral 8x7B model) via LangChain
  • Web Search: Tavily Search API via @langchain/community
  • Authentication: JWT (JSON Web Tokens)
  • Email: Nodemailer with Gmail OAuth2

3. AI FILE EXPLANATIONS (with code walkthrough)
================================================================================

───────────────────────────────────────────────────────────────────────────────
FILE: src/ai/llm.js
PURPOSE: Creates and configures the LLM (Large Language Model) instance
───────────────────────────────────────────────────────────────────────────────

WHAT IT DOES:
  - Imports ChatGroq from @langchain/groq
  - Creates a new ChatGroq instance with your GROQ_API_KEY
  - Configures the model (mixtral-8x7b-32768), temperature (0), and max tokens

CODE EXPLANATION:
  
  import { ChatGroq } from "@langchain/groq";
  // ChatGroq is a class that wraps the Groq API to make it work with LangChain.
  // LangChain needs a standard interface for all LLMs, and ChatGroq provides that.

  export const llm = new ChatGroq({
    apiKey: process.env.GROQ_API_KEY,   // Your API key from Groq console
    model: "mixtral-8x7b-32768",        // Mix of 8 expert models, 32K context window
    temperature: 0,                      // 0 = deterministic (same input = same output)
    maxTokens: 1024,                     // Max response length (prevents runaway responses)
  });

WHY MIXTRAL 8x7B?
  - It's free on Groq
  - 32,768 token context window (can read long documents)
  - Very fast inference on Groq's LPU hardware
  - Supports tool-calling (needed for our agent to use web search)

COMMON ERRORS:
  Error: "GROQ_API_KEY is missing in .env file"
    → You forgot to add your Groq API key. Get one at https://console.groq.com

  Error: "401 Unauthorized"
    → Your GROQ_API_KEY is invalid or expired. Generate a new one.

  Error: "model_not_found"
    → The model name changed. Check Groq's docs for current model names.


───────────────────────────────────────────────────────────────────────────────
FILE: src/ai/tools.js
PURPOSE: Defines the tools (capabilities) available to the AI agent
───────────────────────────────────────────────────────────────────────────────

WHAT IT DOES:
  - Creates a TavilySearchResults tool that searches the web
  - The agent can CHOOSE to use this tool when a question needs current info

CODE EXPLANATION:

  import { TavilySearchResults } from "@langchain/community/tools/tavily_search";
  // TavilySearchResults is a LangChain "tool" that wraps the Tavily web search API.
  // It's imported from @langchain/community (community-maintained tools package).

  export const tools = [
    new TavilySearchResults({
      apiKey: process.env.TAVILY_API_KEY,  // Your Tavily API key
      maxResults: 3,                        // Return top 3 search results
    }),
  ];

HOW TOOLS WORK IN LANGCHAIN:
  - A "tool" is a function that the AI can call
  - The LLM sees the tool's name and description
  - When the LLM decides it needs web info, it outputs a "tool call"
  - LangChain executes the tool and feeds the result back to the LLM
  - The LLM then uses the search results to form its answer

  Example flow:
    User: "What's the weather in Delhi?"
    LLM thinks: "I need current info → use tavily_search"
    Tool call: tavily_search("weather Delhi today")
    Tool result: "Delhi: 32°C, partly cloudy..."
    LLM: "The current weather in Delhi is 32°C and partly cloudy."

COMMON ERRORS:
  Warning: "TAVILY_API_KEY is missing in .env"
    → Web search won't work, but the AI will still answer from its training data.

  Error: "Tavily API rate limit exceeded"
    → Free tier has 1000 searches/month. Upgrade or wait for the next month.


───────────────────────────────────────────────────────────────────────────────
FILE: src/ai/agent.js
PURPOSE: Creates the LangChain agent that combines the LLM with tools
───────────────────────────────────────────────────────────────────────────────

WHAT IT DOES:
  - Defines the system prompt (instructions for the AI)
  - Creates a "tool-calling agent" that can decide when to use tools
  - Wraps it in an AgentExecutor that manages the agent's execution loop

CODE EXPLANATION:

  import { ChatPromptTemplate } from "@langchain/core/prompts";
  // ChatPromptTemplate structures the conversation format for the LLM.

  import { AgentExecutor, createToolCallingAgent } from "langchain/agents";
  // createToolCallingAgent: creates an agent that uses the LLM's native tool-calling
  // AgentExecutor: runs the agent in a loop (ask → tool call → observe → answer)

  const prompt = ChatPromptTemplate.fromMessages([
    ["system", "You are a helpful AI assistant..."],  // AI's personality/instructions
    ["human", "{input}"],                              // User's question placeholder
    ["placeholder", "{agent_scratchpad}"],              // Where tool results go
  ]);

  // CRITICAL: We use createToolCallingAgent, NOT createOpenAIFunctionsAgent
  // createOpenAIFunctionsAgent only works with OpenAI models
  // createToolCallingAgent works with ANY LLM that supports tool-calling (Groq does)
  const agent = await createToolCallingAgent({ llm, tools, prompt });

  const executor = new AgentExecutor({
    agent,
    tools,
    verbose: process.env.NODE_ENV !== "production",  // Log agent thinking in dev
    maxIterations: 5,          // Prevent infinite loops (max 5 tool calls per question)
    handleParsingErrors: true, // Don't crash on malformed LLM output
  });

THE AGENT EXECUTION LOOP:
  1. User sends a question
  2. LLM reads the question + system prompt
  3. LLM decides: "I need to search" or "I can answer directly"
  4. If search needed:
     a. LLM outputs a tool call (tool name + arguments)
     b. AgentExecutor runs the tool
     c. Tool result is added to agent_scratchpad
     d. LLM reads the results and formulates an answer
  5. If no search needed:
     a. LLM directly outputs the answer
  6. AgentExecutor returns the final answer

CRITICAL DESIGN DECISION (createToolCallingAgent vs createOpenAIFunctionsAgent):
  ┌─────────────────────────────────────────────────────────────────────┐
  │  ❌ OLD (BROKEN):  createOpenAIFunctionsAgent                      │
  │     - Uses OpenAI's "function calling" format                      │
  │     - Only works with OpenAI models (GPT-3.5, GPT-4)              │
  │     - Groq's Mixtral does NOT support this format                  │
  │     - Result: Agent crashes on every request                       │
  │                                                                     │
  │  ✅ NEW (FIXED):   createToolCallingAgent                          │
  │     - Uses the standard "tool calling" format                      │
  │     - Works with ANY LLM that supports tool-calling               │
  │     - Groq's Mixtral DOES support this format                      │
  │     - Result: Agent works correctly                                │
  └─────────────────────────────────────────────────────────────────────┘

COMMON ERRORS:
  Error: "Agent creation failed: Check your GROQ_API_KEY and TAVILY_API_KEY"
    → One of the API keys is missing or invalid.

  Error: "Max iterations reached"
    → The agent is stuck in a loop. Increase maxIterations or simplify the question.


───────────────────────────────────────────────────────────────────────────────
FILE: src/ai/ai.service.js
PURPOSE: Public interface to the AI system (the only file other code calls)
───────────────────────────────────────────────────────────────────────────────

WHAT IT DOES:
  - initAI(): Creates the agent once at server startup
  - askAI(message): Sends a question to the agent and returns the answer

CODE EXPLANATION:

  let agent;  // Singleton — created once, reused for all requests

  export const initAI = async () => {
    try {
      agent = await createAgent();
      // If this succeeds, the AI is ready
    } catch (error) {
      agent = null;
      // Server still starts, but AI features are disabled
      // This prevents one bad API key from killing the entire server
    }
  };

  export const askAI = async (message) => {
    if (!agent) throw new Error("AI agent is not initialized");
    
    const response = await agent.invoke({ input: message });
    return response.output;
    // response.output is the final text answer from the agent
  };

WHY IS THIS A SEPARATE FILE?
  - Separation of concerns: controllers don't need to know about LangChain
  - Singleton pattern: agent is created once, not per-request
  - Error isolation: if AI breaks, only this layer handles it
  - Easy to swap: if you switch from Groq to OpenAI, only change ai/ files

COMMON ERRORS:
  Error: "AI agent is not initialized"
    → initAI() failed at startup. Check server logs for the root cause.

  Error: "AI service is temporarily rate-limited"
    → Too many requests to Groq. Wait a few seconds and retry.

  Error: "AI authentication failed"
    → GROQ_API_KEY is invalid. Generate a new one at console.groq.com.

4. HOW THE AI PIPELINE WORKS (step-by-step flow)
================================================================================

  User's Browser                    Backend Server
  ──────────────                    ──────────────
  
  1. User types: "What is React?"
          │
          ▼
  2. POST /api/chats
     Body: { "message": "What is React?" }
          │
          ▼
  3. chatRoutes.js → chatHandler (chat.controller.js)
          │
          ▼
  4. chatHandler validates input:
     - Is message present? (not null/undefined)
     - Is it a string?
     - Is it non-empty?
     - Is it under 5000 chars?
          │
          ▼
  5. chatHandler calls askAI("What is React?")
          │
          ▼
  6. ai.service.js → agent.invoke({ input: "What is React?" })
          │
          ▼
  7. AgentExecutor sends the question to Groq's Mixtral LLM
          │
          ▼
  8. LLM DECIDES: Does this need a web search?
     ┌─── YES (needs current info) ──── NO (can answer from knowledge) ───┐
     │                                                                      │
     ▼                                                                      ▼
  9a. LLM outputs tool call:          9b. LLM directly outputs answer:
      tavily_search("React library")       "React is a JavaScript library..."
     │                                                                      │
     ▼                                                                      │
  10. Tavily searches the web                                               │
      Returns top 3 results                                                 │
     │                                                                      │
     ▼                                                                      │
  11. Results fed back to LLM                                               │
     │                                                                      │
     ▼                                                                      │
  12. LLM formulates final answer                                           │
      using search results                                                  │
     │                                                                      │
     └──────────────────── Both paths converge ────────────────────────────┘
          │
          ▼
  13. Response sent back to user:
      { "success": true, "answer": "React is a JavaScript library for..." }


STARTUP SEQUENCE (when you run `npm run dev`):
  
  ┌──────────────────────────────────────────────────────────────────┐
  │  Step 1: dotenv loads .env variables                             │
  │  Step 2: connectDb() → MongoDB connects                         │
  │  Step 3: initAI() → LangChain agent created (LLM + Tavily)      │
  │  Step 4: app.listen(PORT) → Express server starts                │
  │                                                                   │
  │  If Step 2 fails → Server exits (no database = no service)       │
  │  If Step 3 fails → Server starts, but /api/chats returns 503     │
  │  If Step 4 fails → Server exits (port already in use?)           │
  └──────────────────────────────────────────────────────────────────┘

6. COMMON ERRORS & TROUBLESHOOTING
================================================================================

── 6.1 SERVER WON'T START ─────────────────────────────────────────────────────

  SYMPTOM: Server crashes immediately on startup
  
  Error: "MONGODB_URI is missing in .env file"
  FIX:   Add MONGODB_URI=mongodb://localhost:27017/perplexity to .env

  Error: "GROQ_API_KEY is missing in .env file"
  FIX:   Add your Groq API key to .env

  Error: "MongoDB connection failed: ECONNREFUSED"
  FIX:   MongoDB is not running. Start it with: mongod
         Or if using MongoDB Atlas, check your connection string.

  Error: "EADDRINUSE: address already in use :::3000"
  FIX:   Another process is using port 3000.
         - Windows: netstat -ano | findstr :3000, then taskkill /PID <pid> /F
         - Or change PORT in .env to 3001

── 6.2 AI DOESN'T RESPOND ─────────────────────────────────────────────────────

  SYMPTOM: POST /api/chats returns error

  Error: "AI agent is not initialized"
  FIX:   Check server startup logs for the root cause.
         Usually: missing or invalid GROQ_API_KEY.

  Error: "AI service is temporarily rate-limited"
  FIX:   Wait 10-30 seconds and try again.
         Groq free tier has rate limits.

  Error: "AI authentication failed"
  FIX:   Your GROQ_API_KEY is invalid or expired.
         Go to console.groq.com and create a new key.

  Error: "Max iterations reached"
  FIX:   The question caused the agent to loop.
         Try rephrasing the question. If persistent, increase maxIterations
         in agent.js (currently set to 5).

── 6.3 EMAILS NOT SENDING ─────────────────────────────────────────────────────

  SYMPTOM: Registration works but no verification email received

  Error: "Email transporter is not configured"
  FIX:   Check Google OAuth2 credentials in .env

  Error: "invalid_grant"
  FIX:   Your Google refresh token expired. 
         Generate a new one at the Google OAuth2 Playground:
         https://developers.google.com/oauthplayground

  Error: "Mail transporter verification failed"
  FIX:   This is a warning, not fatal. Check all 4 Google env vars.

── 6.4 AUTHENTICATION ERRORS ──────────────────────────────────────────────────

  Error: "Unauthorised access - No token provided"
  FIX:   You need to login first. The token is stored in a cookie.
         Make sure your frontend sends credentials: true in fetch/axios.

  Error: "Unauthorised access - Invalid token"
  FIX:   The JWT has expired or been tampered with. Login again.

  Error: "Please verify your email before logging in"
  FIX:   Check your email for the verification link.
         If not received, check spam folder or resend.

── 6.5 DATABASE ERRORS ─────────────────────────────────────────────────────────

  Error: "User with this email or username already exists"
  FIX:   Use a different email/username, or login if you already registered.

  Error: "Cast to ObjectId failed"
  FIX:   An invalid MongoDB ID was passed. This is a code bug — 
         validate IDs before using them in queries.

7. DEVELOPER GUIDE: What You Must Know
================================================================================

── 7.1 ADDING A NEW AI TOOL ───────────────────────────────────────────────────

  To give the AI a new capability (e.g., calculator, Wikipedia):

  Step 1: Create or import the tool in tools.js:
    import { Calculator } from "@langchain/community/tools/calculator";
    
    export const tools = [
      new TavilySearchResults({ ... }),
      new Calculator(),  // ← Add new tool to the array
    ];

  Step 2: That's it! The agent automatically discovers tools in the array.
          LangChain reads each tool's name and description and tells the LLM.

── 7.2 CHANGING THE AI MODEL ──────────────────────────────────────────────────

  To switch from Mixtral to another model:

  In llm.js, change the model name:
    export const llm = new ChatGroq({
      model: "llama-3.3-70b-versatile",  // or any Groq-supported model
    });

  Available models on Groq: https://console.groq.com/docs/models

── 7.3 CUSTOMIZING THE AI'S BEHAVIOR ──────────────────────────────────────────

  The system prompt in agent.js controls the AI's personality:

    ["system", "You are a helpful AI assistant similar to Perplexity..."],

  Change this string to modify behavior:
    - "You are a coding assistant. Only answer programming questions."
    - "You are a customer support bot for an e-commerce store."
    - "Always respond in Hindi."

── 7.4 SAVING CHAT HISTORY TO DATABASE ────────────────────────────────────────

  The models (chat.model.js, msg.model.js) are defined but NOT used yet.
  To persist chats:

  In chat.controller.js, after getting the AI response:
    
    const chat = await chatModel.create({ user: req.user.id, title: message });
    await messageModel.create({ chat: chat._id, content: message, role: "user" });
    await messageModel.create({ chat: chat._id, content: reply, role: "ai" });

── 7.5 ERROR HANDLING ARCHITECTURE ────────────────────────────────────────────

  Errors are handled at FOUR levels:

  Level 1: Input Validation (validator/auth.validation.js)
    → Catches bad input BEFORE it reaches the controller
    → Returns 400 with validation errors

  Level 2: Controller try-catch (controllers/*.js)
    → Catches business logic errors (user not found, wrong password)
    → Returns appropriate status codes (400, 401, 404, 500)

  Level 3: Global Error Handler (app.js - last middleware)
    → Catches ANY error that slipped through controllers
    → Returns 500 with generic message (hides internals in production)

  Level 4: Process-level Handlers (server.js)
    → uncaughtException: catches fatal sync errors → logs and exits
    → unhandledRejection: catches forgotten awaits → logs (no exit)
    → SIGTERM/SIGINT: graceful shutdown on kill signal

11. SECURITY CONSIDERATIONS
================================================================================

  ✅ WHAT'S ALREADY SECURE:
    • Passwords are hashed with bcrypt (10 salt rounds)
    • JWT tokens have 1-day expiry
    • Cookies are httpOnly (can't be read by JavaScript/XSS)
    • Cookies are secure in production (HTTPS only)
    • SameSite cookie attribute prevents CSRF
    • CORS restricts which domains can call the API
    • API keys are in .env (not committed to git)

  ⚠️  THINGS TO WATCH:
    • NEVER commit .env to git (add to .gitignore)
    • Use strong JWT_SECRET (at least 32 random hex characters)
    • Rate-limit the /api/chats endpoint to prevent abuse
    • Add auth middleware to /api/chats if you want only logged-in users
    • Set CORS origin to your actual frontend domain in production
